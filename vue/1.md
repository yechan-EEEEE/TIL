## **1. 프론트엔드 개발 (Frontend Development)**

### **프론트엔드 개발이란?**

* 웹사이트의 **사용자 인터페이스(UI)와 사용자 경험(UX)을 만들고 디자인**하는 일.  
* **HTML, CSS, JavaScript** 등을 활용하여 사용자가 서버와 상호작용하는 부분을 개발. (Client ↔ Server)  
* 주로 **Frontend Framework** (Vue, React, Angular 등)를 사용하여 개발.

### **Client-side frameworks**

#### **Client-side frameworks가 필요한 이유**

1. **데이터 변화에 대한 DOM 요소의 변경 처리**:  
   * 서버에서 데이터를 받아와 DOM 요소를 수정하는 **AJAX 방식**은 데이터를 처리할 때마다 브라우저가 DOM 요소를 직접 찾아 수정해야 하므로 번거롭고 많은 코드를 필요로 함.  
   * Client-side frameworks는 **데이터가 변하면 화면이 바뀌는 Vue의 반응형(Reactivity) 시스템**을 통해 이 과정을 자동화하여, 더 **간편한 개발**을 가능하게 함.  
2. **대규모 애플리케이션 개발 용이성**:  
   * 현대적인 웹 애플리케이션은 **복잡하고 대화형**이며, 사용자 상호작용이 많은 기능을 포함함.  
   * Client-side frameworks는 복잡한 애플리케이션을 개발하고 관리하기 위한 **구조와 도구**를 제공하여, 개발자가 더 쉽게 기능 구현에 집중할 수 있도록 함. (Vue, React, Angular가 대표적)

### **SPA (Single Page Application)**

* **단일 페이지**에서 동작하는 애플리케이션.  
* 최초 HTML 파일 위에서 **JavaScript가 필요한 부분만 리소스**를 다운로드.  
* 페이지 갱신 시, 필요한 데이터만 **비동기적**으로 서버에 전송받아 화면의 **일부 DOM만 동적**으로 업데이트.  
* **페이지 전체를 새로고침**할 필요가 없어 빠르고 끊김 없는 **사용자 경험**을 제공.  
* **JavaScript를 사용**하여 클라이언트 측에서 콘텐츠를 생성하고 렌더링하는 방식 (== **CSR**).

### **CSR (Client Side Rendering)**

* 클라이언트(브라우저)에서 **콘텐츠를 렌더링**하는 방식.  
* 서버는 최소한의 **빈 HTML**과 JavaScript 파일을 클라이언트에 전달.  
* 클라이언트의 **JavaScript가 실행**되면서 데이터를 가져오고 DOM을 조작하여 페이지 콘텐츠를 렌더링.  
* **CSR의 장점**: 서버 부하 감소, 빠른 페이지 전환 속도, 사용자 경험 향상.  
* **CSR의 단점**: 초기 로딩 속도 느림, SEO에 불리함 (검색엔진이 HTML을 파싱하지만, 콘텐츠가 JavaScript로 생성되기 때문).

## **2. Vue**

### **Vue 개요**

* 웹사이트를 **쉽고 빠르게** 만들 수 있도록 돕는 **프론트엔드 JavaScript 프레임워크**.  
* **점진적 채택** (Block-based) 및 **반응형(Reactive)** 특징을 가짐.  
* **배우기 쉽고** 직관적이어서 초보자부터 전문가까지 폭넓게 사용됨.  
* **Evan You**에 의해 2014년 발표. (Angular, React 경험을 바탕으로 개발)

### **Vue 학습의 이점**

* **낮은 학습 곡선**: 직관적, 간결하여 비교적 쉽게 배우고 익힐 수 있음.  
* **확장성 및 생산성**: 작고 가벼운 라이브러리부터 대규모 애플리케이션까지 유연하게 확장 가능.  
* **유연한 프로젝트 진행**: 규모가 작은 프로젝트부터 대규모 데모 애플리케이션까지 적용 가능.

### **Vue의 2가지 핵심 기능**

1. **선언적 렌더링 (Declarative Rendering)**  
   * **템플릿 구문**을 사용하여 JavaScript 데이터(상태)를 기반으로 HTML을 화면에 출력.  
   * **{{ message }}**와 같은 템플릿 문법을 사용.  
2. **반응성 (Reactivity)**  
   * JavaScript 상태(데이터)가 변경되면 변경 사항을 추적하여 **자동으로 DOM이 업데이트**됨.  
   * 개발자가 직접 DOM을 조작할 필요 없이 **데이터 변경에만 집중**할 수 있음.

### **Component**

* **재사용 가능한 코드 블록**으로, UI를 독립적인 조각으로 분할하여 개별적으로 개발.  
* Vue 앱은 여러 개의 Component를 트리 형태로 구성하여 만듦.  
  * 예: Header, Main, Article, Footer 등의 컴포넌트로 웹페이지를 구성.

## **3. Vue Tutorial**

### **Vue를 사용하는 방법**

1. **CDN (Content Delivery Network) 방식**: 간단한 테스트나 학습 시 적합.  
2. **NPM (Node Package Manager) 방식**: 대규모 프로젝트 개발 시 적합.  
   * *본 교육 과정에서는 CDN 방식 이후 NPM 방식으로 진행.*

### **Vue Application 생성**

* Vue 애플리케이션은 **createApp** 함수로 생성하며, 앱 인스턴스를 반환.  
* **setup()** 함수는 컴포넌트(루트 컴포넌트)에 대한 정보를 정의하며, 데이터와 함수를 준비하는 역할을 함.  
* **app.mount('#app')**을 사용하여 Vue 애플리케이션을 지정된 DOM 요소(id="app")에 연결(마운트).

JavaScript
```
const { createApp, ref } = Vue;

const app = createApp({  
  setup() {  
    const message = ref('Hello vue!'); // 반응형 상태 선언  
    return {  
      message  
    }  
  }  
});

app.mount('#app'); // 'app'이라는 id를 가진 DOM 요소에 마운트
```
### **반응형 상태 (ref)**

* **ref** 는 Vue에서 **반응형 데이터(상태)**를 선언하는 함수.  
  * ref를 사용하여 선언된 변수는 값이 변경되면 Vue의 **반응형 시스템**이 감지하여 화면을 자동으로 업데이트함.  
* 일반 JavaScript 변수와 달리, ref로 생성된 객체의 **.value** 속성에 실제 값이 저장됨.  
  * 예: const count = ref(0); -> 실제 값은 count.value로 접근.  
* **템플릿 내부**에서는 **.value 없이** 바로 count로 사용할 수 있음. (== **Ref Unwrap**)

### **Vue 기본 구조 및 템플릿 렌더링**

* **<script> 블록**: Vue 인스턴스를 생성하고 상태(데이터)와 로직을 정의. (setup() 함수 내부)  
* **<template> (HTML) 블록**: Vue의 상태를 화면에 렌더링.  
  * **Mustache syntax** ({{ expression }})를 사용하여 데이터를 동적으로 렌더링.

HTML
```
<div id="app">  
  <h1>{{ message }}</h1>   
</div>
```
### **ref 객체와 Ref Unwrap 주의사항**

* **ref가 필요한 이유**:  
  * JavaScript는 변수 값의 변경을 추적할 수 없음.  
  * ref는 객체 형태로 값을 래핑하여 **값의 변경**이 발생할 때마다 Vue의 **반응형 시스템**이 이를 감지하도록 함.  
* **Ref Unwrap 주의사항**:  
  * ref는 템플릿에서 .value 없이 사용되지만, **다른 반응형 객체의 속성으로 사용**될 때는 unwrap 되지 않음.  
  * 객체 내부에서 const object = ref({ id: ref(0) }) 와 같이 사용하면 **object.id가 아닌 object.id.value** 로 접근해야 함.

## **4. SEO & CSR/SSR 비교**

### **SEO (Search Engine Optimization)**

* **검색 엔진 최적화**; Google, Naver 등 검색 엔진 결과에 잘 노출되도록 개선하는 작업.  
* **CSR의 문제점**: 초기 HTML이 비어있어, 검색 엔진이 콘텐츠를 파악하기 어려워 SEO에 불리함.  
  * 검색 엔진의 JS 실행 능력이 향상되고 있지만, 여전히 SSR이 SEO에 더 유리한 경우가 많음.  
* **SPA 기반의 CSR 단점 보완**: **Nuxt.js** 와 같은 **하이브리드 프레임워크**를 사용하여 SSR(Server Side Rendering) 기능을 도입해 SEO 문제점을 극복할 수 있음.

### **CSR vs. SSR (Client Side Rendering vs. Server Side Rendering)**

| 특징 | CSR (Client Side Rendering) | SSR (Server Side Rendering) |
| :---- | :---- | :---- |
| **렌더링 주체** | 클라이언트 (브라우저) | 서버 |
| **초기 로딩** | 빈 HTML 전송 후 JS 실행/렌더링 → 느릴 수 있음 | 완성된 HTML 전송 → 빠름 |
| **페이지 이동** | 비동기 업데이트 (SPA) → 빠름 | 서버에서 새 페이지 렌더링 → 느릴 수 있음 |
| **SEO** | 불리함 (JS 실행 필요) | 유리함 (완성된 HTML 제공) |
